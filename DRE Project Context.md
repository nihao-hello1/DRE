

你现在要协助我开发一个项目，名称暂定为 **DRE（Document Rendering Engine）**。

## 一、项目目标

DRE 的目标是解决 **AI Agent 最后一公里文档交付问题**。

现在的常见流程是：

1. 用户让 Agent 写一个长文档，例如技术方案、网络改造方案、系统设计方案。
2. Agent 通常先生成 Markdown 或类似的结构化文本。
3. 用户多轮修改后，内容确认无误。
4. 最后需要把内容渲染成 **专业可交付的 Word 文档（DOCX）**。
5. 现实中这一步通常要人工调整排版，成本高、易出错、格式不统一。

DRE 要做的事是：

> 在用户确认内容后，把最终的 Markdown / 文档结构，自动渲染成格式规范、缩进、目录序号层级规范美观、可直接交付的 DOCX。

------

## 二、项目边界

### DRE 负责什么

- Markdown / 文档结构解析
- 文档结构标准化
- 标题层级映射
- 字体、字号、行距、缩进、段距
- 自动目录
- 页眉页脚
- 页码
- 表格样式
- 图片样式
- Caption / 图表编号
- DOCX 生成
- Word 后处理
- 专业文档模板应用

### DRE 不负责什么

- 内容创作
- 内容润色
- 文档审稿
- 需求讨论
- 多轮对话写作
- 选题
- 自媒体内容生成

这些都由 Agent 负责。

DRE 只负责：

> **内容确认之后的文档渲染与交付。**

------

## 三、核心原则

### 1. 内容与排版分离

Agent 负责内容，DRE 负责排版和输出。

### 2. 模板驱动

不要让模型自己“猜” Word 怎么排版，必须由模板和规则决定。

### 3. Agent 无关

DRE 不是某一个 Agent 的专属功能，必须尽量兼容主流 Agent，例如：

- Claude Code
- Codex
- Hermes
- OpenClaw
- Trae
- 其他支持 Skills / MCP / 类似扩展机制的 Agent

### 4. 最终输出必须是 DOCX

Markdown 只是一种内部中间格式，不是最终交付格式。

### 5. 先做一个文档类型

第一版只做一种类型，先把一个场景做透，后面再扩展。

优先建议第一版只支持：

> **技术设计方案 / 技术文档类长文档**

例如网络改造技术方案。

------

## 四、工作流

推荐工作流如下：

1. 用户通过 Agent 写文档。
2. Agent 生成 Markdown 或文档结构。
3. 用户反复修改，直到确认内容没问题。
4. 当文档进入“可渲染”状态时，Agent 提醒用户可以执行最终渲染。
5. DRE 将 Markdown / 文档结构转换为规范 DOCX。
6. 输出最终可交付 Word 文件。

也就是说：

> Agent 负责“写对内容”，DRE 负责“把内容变成正式文档”。

------

## 五、DRE 的职责拆分

### 1. Parser

把 Markdown / 文档结构解析成统一的中间表示。

### 2. AST / Document Model

定义统一的文档结构，例如：

- title
- heading
- paragraph
- list
- table
- image
- quote
- code block

### 3. Style Engine

把文档语义映射成固定的 Word 样式规则，例如：

- Heading 1 / 2 / 3 的字体和字号
- 正文行距
- 段前段后
- 首行缩进
- 表格样式
- 图片说明样式

### 4. DOCX Renderer

把结构化内容渲染为 DOCX。

### 5. Office Post-Processing

如果需要，使用 OfficeCLI 或类似工具做 Word 后处理，例如：

- 更新目录
- 修复样式
- 统一格式
- 补齐页眉页脚
- 调整编号

------

## 六、Agent 接入方式

DRE 需要兼容两类入口：

### 1. Skills / Rules / Prompt Pack

用于让 Agent 认识这个项目，知道何时提醒用户进入最终渲染阶段，知道应该如何准备文档结构。

### 2. MCP

用于提供真实能力，例如：

- render_document
- validate_document
- apply_template
- export_docx
- update_toc

Skill 负责“告诉 Agent 怎么用”，MCP 负责“真正执行”。

------

## 七、交互体验要求

DRE 需要支持一个很自然的交互状态流：

- Draft
- Developing
- Reviewing
- Ready for Rendering
- Rendered

当用户已经多轮修改，并明确表达“可以了”“没问题”“定稿”“生成正式版”时，Agent 应该提醒：

> 当前文档已可进入最终渲染，是否生成正式 DOCX？

不要让用户自己猜什么时候该导出。

------

## 八、第一版 MVP

第一版不要做大而全。

### MVP 输入

- Markdown
- 或者 Agent 输出的结构化文档内容

### MVP 输出

- 一个规范的 DOCX

### MVP 必须包含

- 标题层级
- 正文样式
- 自动目录
- 页码
- 页眉页脚
- 基础表格样式
- 图片样式
- 基础编号
- 基础模板能力

### MVP 暂不做

- 多文档类型
- 复杂企业模板库
- 招投标全流程
- 学术论文全套规范
- 复杂内容创作能力

## 十一、成功标准

当我让 Agent 写一个技术方案，并经过多轮确认后，DRE 能自动把最终内容渲染成：

- 结构清晰
- 排版统一
- 可以直接交付
- 无需人工再手动大改的 DOCX

这就算第一阶段成功。
